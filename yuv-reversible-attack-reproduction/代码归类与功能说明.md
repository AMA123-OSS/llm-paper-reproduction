# 代码归类与功能说明

本文档用于说明当前 GitHub 仓库中已经上传的代码文件分别承担什么功能。为避免混淆，先说明整体结构：

- 根目录下的 `.py` 文件是论文原始复现主代码和核心算法模块。
- `runs/` 下主要是分阶段复现实验脚本、评估脚本和实验记录。
- `update_yuv_recovery_optimization/` 下是后续改进实验代码，重点对应可逆恢复闭环、UV嵌入失真分析和迁移性扩展。
- `pytorch_grad_cam/` 下是本项目使用的本地 Grad-CAM 辅助代码。
- `side_by_side_reproduction/` 是为了在 GitHub 上并列展示“原始复现代码/结果”和“更新优化代码/结果”的副本目录，不是新的算法实现来源。

## 一、原始论文复现核心代码

位置：`yuv-reversible-attack-reproduction/`

这部分代码对应论文主方法，是最核心的复现代码。

| 文件 | 作用 | 主要功能 |
|---|---|---|
| `reversible_attack_OURS.py` | 原始主流程脚本 | 串联图像读取、模型加载、CAM显著区域提取、Y通道攻击、UV通道恢复信息嵌入、RAE保存和指标统计，是原始论文方法的主入口。 |
| `atk.py` | 对抗攻击方法实现 | 实现 RGB/YUV 空间下的 FGSM、BIM、PGD、MI-FGSM、NI-FGSM 等攻击方法，支持指定通道攻击，论文复现中主要用于 Y 通道扰动生成。 |
| `EnModel.py` | 集成模型定义 | 定义由 DenseNet161、InceptionV3、GoogLeNet 组成的 EnModel，对多个模型 logits 取平均，用于提升扰动迁移性。 |
| `embed_utils.py` | 压缩与PEE嵌入核心 | 包含 `compressfun`、预测误差扩展 PEE 编码/解码、阈值选择和原始 `embed_main`，负责把 Y 通道差分信息嵌入 U/V 通道。 |
| `helpers.py` | 算术编码工具 | 提供 Arithmetic Encoding / Decoding，用于压缩和恢复 Y 通道差分 bitstream。 |
| `utils.py` | RGB/YUV颜色空间转换 | 实现 `rgb2yuv` 和 `yuv2rgb`，负责在 RGB 图像和 YUV 表示之间转换。 |
| `calMetrics.py` | 图像质量指标计算 | 计算 SSIM、PSNR、L2、Linf 等指标，用于衡量生成 RAE 与原图之间的视觉差异。 |

### 原始主流程关系

原始复现主流程可理解为：

```text
输入RGB图像
  -> utils.py 转为YUV
  -> atk.py + EnModel.py 在Y通道生成扰动
  -> pytorch_grad_cam/ 生成CAM显著区域
  -> embed_utils.py 压缩Y差分并嵌入U/V
  -> utils.py 转回RGB得到RAE
  -> calMetrics.py 计算图像质量指标
```

## 二、Grad-CAM辅助代码

位置：`yuv-reversible-attack-reproduction/pytorch_grad_cam/`

这部分代码用于生成显著性区域，辅助论文方法减少无效扰动区域。

| 文件 | 作用 | 主要功能 |
|---|---|---|
| `gradcam.py` | CAM主实现 | 根据模型中间层梯度和激活生成 Grad-CAM / Grad-CAM++ 热力图。 |
| `activations_and_gradients.py` | 中间特征和梯度捕获 | 注册 hook，保存目标层的 activation 和 gradient，为 CAM 计算提供输入。 |
| `guided_backprop.py` | Guided Backprop 辅助 | 提供 guided backpropagation 相关功能，用于可视化或辅助解释。 |
| `utils/image.py` | 图像工具函数 | 提供 CAM 结果归一化、叠加或图像处理相关工具。 |
| `__init__.py`、`utils/__init__.py` | 包初始化 | 使该目录可以作为 Python 模块导入。 |

在本项目中，CAM 的核心作用不是最终展示图片，而是确定哪些区域对分类更重要，从而约束 Y 通道扰动区域，减少 GRP 问题。

## 三、原始分阶段复现实验代码

位置：`yuv-reversible-attack-reproduction/runs/`

`runs/` 目录保存的是围绕原始代码逐步复现、评估、消融和总结的脚本。它们不是论文原始源码，而是为了让复现实验可追踪、可复查而写的实验脚本。

| 文件 | 对应阶段 | 作用 |
|---|---|---|
| `evaluate_outputs.py` | 阶段6 | 通用输出评估脚本，对生成结果计算 SSIM、PSNR、L2、Linf 等指标。 |
| `stage7_evaluate_runs.py` | 阶段7 | 对 10 张图小规模复现结果做逐图评估和汇总。 |
| `stage8_ablation_experiment.py` | 阶段8 | 执行 RGB MI-FGSM、Y-only、Y+UV embed 三组消融实验，比较攻击成功率和图像质量。 |
| `stage9_transferability_experiment.py` | 阶段9 | 执行迁移性实验，比较 single Inception 与 EnModel 在不同测试模型上的成功率。 |
| `stage10_recovery_feasibility_check.py` | 阶段10 | 检查原始恢复流程的前置条件，分析为什么公开代码还不能直接完成完整无误差恢复。 |
| `final_reproduction_summary.py` | 阶段11 | 汇总阶段性结果，生成最终复现摘要 CSV/Markdown。 |

### runs目录中的结果文件

`runs/` 中的 `.csv` 和 `.md` 主要是结果记录，不是代码：

- `stage7_10img_summary.csv`：10图小规模复现汇总。
- `stage8_ablation_summary.csv`：Y/UV消融实验汇总。
- `stage9_transferability_summary.csv`：迁移性实验汇总。
- `stage10_recovery_preconditions.csv`：恢复闭环前置条件检查。
- `final_reproduction_summary.md`：最终复现实验摘要。
- `阶段*.md`：分阶段代码阅读、测试、消融、迁移性和恢复验证记录。

这些文件的作用是保留“实验怎么跑、跑出了什么、结论是什么”，方便别人复查实验过程。

## 四、后续更新优化代码

位置：`yuv-reversible-attack-reproduction/update_yuv_recovery_optimization/`

这部分是后续针对原始复现不足进行的改进实验代码。它与原始代码隔离，目的是不破坏原始复现流程，同时补充更清楚的实验验证。

| 文件 | 对应路线 | 作用 |
|---|---|---|
| `recovery_embed.py` | 路线A | 新增带元数据的压缩、嵌入和恢复函数。相比原始 `embed_utils.py`，它保存 `code`、`dic`、`compressed_bit_length`、每轮 `T` 和 payload 长度，从而支持完整恢复闭环。 |
| `recover_from_rae.py` | 路线A | 命令行恢复工具，根据 `rae_yuv.npy` 和 `recovery.json` sidecar 恢复原始 YUV。 |
| `route_a_recovery_selfcheck.py` | 路线A | 构造 10 个合成 YUV 样本，验证 sidecar 补齐元数据后能否做到 `recovery_max_abs_error=0` 和 `recovery_error_pixels=0`。 |
| `route_b_uv_payload_analysis.py` | 路线B | 分析默认 UV 嵌入中的 payload 长度、U/V 改动像素数、UV PSNR 等，用于定位 UV 失真来源。 |
| `route_b_ablation_plan.py` | 路线B | 生成路线B消融实验计划，列出 U-only、V-only、U/V比例、阈值策略、sidecar外置等候选方案。 |
| `route_b_uv_ablation_selfcheck.py` | 路线B | 实际运行合成消融实验，比较 U-only、V-only、U/V split、adaptive_T_cost、sidecar_external_payload 等策略的失真和可逆性。 |
| `route_c_transferability_preflight.py` | 路线C | 检查迁移性扩展实验所需环境，生成 10图/100图、7种生成方式、8个测试模型的 112 行实验矩阵。 |
| `route_c_transferability_experiment.py` | 路线C | 真实迁移性扩展实验脚本。恢复 `ORI_IMG`、torch、torchvision 后，可运行 VGG16、MobileNetV3、EfficientNet、ConvNeXt、ViT 等 held-out 模型测试。 |
| `update_experiment_summary.md` | 总结 | 汇总路线A/B/C的更新结果、环境限制和后续方向。 |
| `改进实验规范报告.md` | 报告 | 按“实验目标和结论、实验设计、实验数据、数据解析”规范说明改进实验为什么能支撑对应结论。 |

### 更新代码的核心逻辑

路线 A 解决的是“能不能真正恢复”：

```text
原始YUV + 攻击后YUV
  -> 计算Y差分
  -> 算术编码压缩
  -> PEE嵌入U/V
  -> 保存recovery.json sidecar
  -> 反向PEE解码
  -> 算术解码Y差分
  -> 恢复原始YUV
```

路线 B 解决的是“UV嵌入为什么失真、怎么降低失真”：

```text
同一payload
  -> 分别测试 U+V / U-only / V-only / U-V比例 / 阈值策略 / sidecar外置
  -> 比较 PSNR、SSIM、L2、Linf、可逆性
```

路线 C 解决的是“迁移性实验如何更规范”：

```text
生成方式：single / full ensemble / leave-one-out
测试模型：ensemble member + held-out black-box
规模：10图 + 100图
指标：成功率 + 条件成功率 + 95%置信区间 + 图像质量
```

## 五、并列展示目录

位置：`yuv-reversible-attack-reproduction/side_by_side_reproduction/`

这个目录是为了 GitHub 展示和老师查看而整理的副本目录。它的作用是让“原始代码/更新代码/原始结果/更新结果”四类内容并列摆放，而不是新增一套独立实验。

| 子目录 | 内容 | 说明 |
|---|---|---|
| `01_original_code/core/` | 原始核心代码副本 | 复制根目录核心 `.py` 文件，方便和更新代码对照。 |
| `01_original_code/experiment_scripts/` | 原始阶段实验脚本副本 | 复制 `runs/` 中的实验脚本。 |
| `02_updated_code/` | 更新优化代码副本 | 复制 `update_yuv_recovery_optimization/` 中的 `.py` 和说明文档。 |
| `03_original_results/` | 原始复现结果副本 | 复制 `runs/` 中的 CSV 和 Markdown 结果。 |
| `04_updated_results/` | 更新优化结果副本 | 复制路线A/B/C的 CSV、JSON 和报告。 |

因此，如果老师只想快速对照，应优先打开：

```text
side_by_side_reproduction/README.md
```

```text
根目录核心代码
runs/
update_yuv_recovery_optimization/
```

## 六、结果文件与代码文件的区别

为避免误解，仓库中并不是所有文件都是代码。

### 代码文件

主要包括：

- 根目录 `.py`
- `runs/*.py`
- `update_yuv_recovery_optimization/*.py`
- `pytorch_grad_cam/**/*.py`

这些文件可以执行或被导入。

### 结果文件

主要包括：

- `.csv`：实验结果表，例如攻击成功率、PSNR、SSIM、恢复误差。
- `.md`：实验记录、阶段报告、规范报告。
- `.json`：恢复 sidecar 元数据，用于记录每张合成样本的恢复参数。

这些文件主要用于复查和报告，不是算法执行入口。

### 未上传的大文件

为了保持仓库轻量，以下内容没有上传：

- `ORI_IMG/` 原始图片数据；
- 生成的 RAE PNG；
- `.npy` 中间数组；
- 模型权重；
- 虚拟环境；
- 缓存文件。

## 七、建议给老师查看的顺序

建议按以下顺序阅读：

1. `README.md`：了解项目整体。
2. `代码归类与功能说明.md`：了解每个代码文件的作用。
3. `side_by_side_reproduction/README.md`：查看原始代码/更新代码/原始结果/更新结果的并列关系。
4. `runs/final_reproduction_summary.md`：查看原始复现实验总结。
5. `update_yuv_recovery_optimization/改进实验规范报告.md`：查看后续改进实验是否能支撑结论。
6. `update_yuv_recovery_optimization/update_experiment_summary.md`：查看路线A/B/C的简要结果和后续限制。

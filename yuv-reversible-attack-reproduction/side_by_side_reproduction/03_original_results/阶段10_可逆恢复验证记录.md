# 阶段 10：可逆恢复验证记录

日期：2026-06-09

## 阶段目标

验证当前本地代码与已保存 RAE 输出是否能完成论文声称的 error-free recovery：从 RAE 中提取嵌入信息，恢复原始图像，并与原图逐像素比较。

## 代码核查

| 项目 | 状态 | 说明 |
| --- | --- | --- |
| `PE_decode` | present | `embed_utils.py` 中存在预测误差扩展的解码函数 |
| `Arithmetic_decode` | present | `helpers.py` 中存在算术解码函数 |
| `T_flag` 持久化 | missing | `embed_main` 内部计算 `T_flag`，但不返回、不保存 |
| 算术编码字典持久化 | missing | `compressfun` 丢弃 `dic`，只返回 code |
| 压缩流恢复元数据 | missing | 已保存 PNG 中没有完整保存恢复所需的字典与阈值序列 |
| 独立恢复入口 | missing | 主脚本只保存 RAE PNG，没有 RAE -> 原图的恢复脚本 |

## 阶段结论

- 当前仓库提供了部分嵌入/解码积木，但没有完整恢复流程。
- 当前主脚本保存的 RAE PNG 不包含恢复所需的全部元数据，尤其是 `T_flag` 和算术编码字典。
- 因此，本地阶段 10 无法证明 error-free recovery，结论应写为：当前公开代码/当前保存方式下，恢复验证被阻断。
- 这不是对论文理论机制的否定，而是对当前代码复现完整性的限制说明。

## 改进方向

- 修改 `compressfun`，同时返回 `code` 和 `dic`。
- 修改 `embed_main`，返回 `img_result`、`T_flag`、算术编码字典、压缩流长度等恢复元数据。
- 将恢复元数据保存到 sidecar 文件，例如 `0001_321.recovery.json`。
- 新增 `recover_from_rae.py`：输入 RAE PNG 和 sidecar 元数据，执行 `PE_decode`、`Arithmetic_decode`、Y 通道恢复和 RGB 重建。
- 再使用阶段 6 的恢复指标字段计算 `recovery_max_abs_error` 和 `recovery_error_pixels`。
